# Analysis of SDXL and WAN 2.2 Pipelines in the WebUI Codex

## Codebase Structure and Pipeline Overview

The repository is organized with a clear separation of concerns: the **apps/backend** folder contains the engine logic and pipelines, while **apps/interface** holds the Vue 3.5 front-end. Within the backend, pipelines are implemented in a modular fashion. The **SDXL** and **WAN 2.2** pipelines each have dedicated engine classes under apps/backend/engines/, along with supporting runtime modules and use-case runners. This design was inspired by Automatic1111 (A1111), the Forge fork, and ComfyUI, attempting to create a unified “Codex” pattern for diffusion tasks.

**SDXL Pipeline Implementation:** The SDXL pipeline is orchestrated by the StableDiffusionXL engine class (apps/backend/engines/sd/sdxl.py). This engine handles model loading and exposes a txt2img method. When a text-to-image request comes in, the engine:

1. **Loads Models:** On initialization, it loads the two text encoders (the CLIP-L and CLIP-G branches), the UNet, VAE, etc., assembling them via assemble\_engine\_runtime with the SDXL\_SPEC. The spec defines two **“classic”** text branches for SDXL: clip\_l (Clip-Large) and clip\_g (Clip-“G” with text projection)[[1]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/sd/spec.py#L307-L316)[[2]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/sd/spec.py#L323-L331). This means the runtime is prepared with both text encoders present.
2. **Builds Conditioning:** In StableDiffusionXL.txt2img, it creates a CodexProcessingTxt2Img object (containing prompt, resolution, steps, etc.), then calls self.get\_learned\_conditioning(prompts) to encode the text. This method runs both encoders:
3. runtime.classic\_engine("clip\_l") for the main text encoder and
4. runtime.classic\_engine("clip\_g") for the second encoder.

The outputs are combined into a **conditioning dict**. The code shows they capture both sequence embeddings and the pooled output (global embedding) from CLIP-G. They then **concatenate** the token embeddings from CLIP-L and CLIP-G along the feature dimension and package the pooled output with some additional metadata (height/width embeddings) into the vector field[[3]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/sd/sdxl.py#L253-L261)[[4]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/sd/sdxl.py#L274-L283). The result is a dictionary cond with keys "crossattn" (the concatenated context of shape [batch, seq\_len, 2048]) and "vector" (the pooled embedding augmented with image size info)[[5]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/sd/sdxl.py#L292-L299). A similar dict is built for the negative prompt (uncond). This design aligns with the SDXL spec, which requires **dual encoder prompts** “packed into conditioning dicts”[[6]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/.sangoi/architecture/model-pipelines-bible.md#L48-L55).

1. **Pipeline Runner Execution:** The engine then delegates to a pipeline runner. It calls \_generate\_txt2img, which is an alias for use\_cases/txt2img.generate\_txt2img. This function wraps the actual execution in a Txt2ImgPipelineRunner (found in apps/backend/use\_cases/txt2img\_pipeline/runner.py). The runner prepares the sampling plan (applying things like Hi-Res fix, tiling, etc.) and ultimately calls execute\_sampling(...) with the processing object and the conditioning\_data and unconditional\_data it got from the engine[[7]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/sd/sdxl.py#L189-L198).
2. **Denoising Loop and Sampler:** The execute\_sampling function (in runtime/workflows/common.py) sets up a CodexSampler for the chosen algorithm (e.g. Euler)[[8]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/runtime/workflows/common.py#L182-L190). It then enters the iterative denoising loop, calling the sampler’s .sample() method. Inside CodexSampler.sample, at each step it prepares the latents and calls the UNet (wrapped by a KModel/UnetPatcher) with the current sigma/noise. Crucially, this sampler uses the **CFG (classifier-free guidance)** technique: it runs the model with the conditional and unconditional embeddings and combines them. In the code, we see that the sampler compiles the conditions via compile\_conditions(cond) and compile\_conditions(uncond)[[9]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/runtime/sampling/driver.py#L146-L154), then for each step does something equivalent to:

# Pseudocode for CFG inside sampling\_function\_inner
eps\_cond = model(x, sigma, cond)
eps\_uncond = model(x, sigma, uncond)
denoised = eps\_uncond + (eps\_cond - eps\_uncond) \* cfg\_scale

The compile\_conditions function shows that it transforms our cond dict into the internal format expected by the model: it creates a **ConditionCrossAttn** for the "crossattn" tensor and a **Condition** for the "vector" (alias 'y')[[10]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/runtime/sampling/condition.py#L100-L108)[[11]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/runtime/sampling/condition.py#L110-L118). These are stored in model\_conds so that the UNet can access them during forward passes. In theory, the UNet’s cross-attention layers should use c\_crossattn (the concatenated text context), and the global vector y should be fed into the model (SDXL’s UNet uses this as an additional conditioning, often in the form of an added embedding to the mid-layer or an ADM-like conditioning vector).

1. **Result Assembly:** After all sampling steps, the pipeline decodes the final latent to an image using the VAE and returns the result.

Overall, the SDXL pipeline code flow is quite complex, with many moving parts (engines, runtime assembly, pipeline runners, sampler, etc.). This complexity may have introduced some integration issues.

## The Text Encoder Injection Issue (SDXL)

The symptom described – *“text encoders not being injected, resulting in a goop of noise”* – indicates that the text conditioning is likely not affecting the image generation. In other words, the model is generating basically random noise as if it were running unconditionally. Given the code above, several potential causes emerge:

* **Conditioning not applied in UNet forward:** Even though cond and uncond are computed, the UNet might not actually be using them. This could happen if the cross-attention context isn’t properly passed into the model. In the current design, after compiling conditions, the code relies on the CodexSampler and underlying model wrapper to handle injecting those conditions. We need to verify that the UnetPatcher/KModel reads the c\_crossattn and y from model\_conds. If there’s a bug here – for example, the UNet’s attention layers not hooked up to use c\_crossattn – the model would ignore the prompt embeddings. The presence of **noise outputs suggests the UNet is likely running with no textual context.** This is the prime suspect.
* **Global text embedding not utilized:** SDXL’s second text encoder output (the pooled vector we label as y) should influence the UNet (in Stable Diffusion XL, this global embedding is used for ADM-like conditioning). The code does package and pass it[[5]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/sd/sdxl.py#L292-L299)[[11]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/runtime/sampling/condition.py#L110-L118), but if the UNet or sampling function doesn’t actually incorporate y, then half of the conditioning is lost. Missing this can severely degrade output quality or make it nonsensical. We should ensure that the StableDiffusionXL runtime or the CodexSampler attaches this y vector to the UNet. For instance, HuggingFace’s SDXL pipeline passes the global text embedding separately (often via an additional\_embeddings or a specific parameter to the UNet). In our code, the attempt was to handle it via the model\_conds['y'], but it’s possible the UNet forward doesn’t pick it up. This would also result in the text prompt having little to no effect.
* **Concatenation strategy issues:** The code concatenates CLIP-L and CLIP-G outputs along the feature dimension to form a single crossattn tensor of size 2048. This assumes the UNet was initialized to expect 2048-dim cross-attention context (which is likely true given SDXL’s architecture). However, if there was any mismatch (say the UNet expecting them in a different form, or lengths not matching), it could silently fail to attend properly. For example, if CLIP-L and CLIP-G had different token counts and the code didn’t pad one to match lengths, the cat operation might misalign the conditioning. The implementation assumes both encoders produce the same sequence length (and in SDXL they typically do share a common token count), so this is less likely but worth double-checking.
* **CFG combination logic**: If compiled\_cond or compiled\_uncond were empty or zeros, CFG would also yield noise. The code does zero-out conditioning for an *empty negative prompt* (to mimic no conditioning)[[12]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/sd/sdxl.py#L283-L291), which is correct. However, if somehow both cond and uncond were identical or both zero (e.g., if the positive prompt wasn’t encoded correctly and ended up blank), the output would be random. It’s worth logging the shapes and stats of cond['crossattn'] to ensure it’s not all zeros except for the negative case. The debugging hooks show some logging of tensor stats during decoding[[13]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/sd/sdxl.py#L324-L333), but not specifically after conditioning.

In summary, the *most likely culprit is a gap in connecting the computed text embeddings to the UNet’s cross-attention.* The code path is non-trivial: the Txt2ImgPipelineRunner doesn’t explicitly pass conditioning into model.forward; instead it relies on the sampler to do so via global state or the model\_conds. If any part of that chain is broken (for example, if the UnetPatcher wasn’t properly patched to use ConditionCrossAttn inputs), the UNet would default to no textual context.

**Recommendation:** Focus on the integration between compile\_conditions and the UNet’s forward pass. Confirm that the UNet (or KModel) uses the provided c\_crossattn and y. In ComfyUI or k-diffusion implementations, typically the cross-attn layers pull from a context variable that gets set (often via something like model.diffusion\_model.set\_cond()). If our Codex implementation missed that step, we need to add it. For instance, ensure that when CodexSampler.sample calls model(..., cond), the cond (which is a list of one dict with model\_conds) is actually fed into the SpatialTransformer layers. We may need to manually set unet.model.cond = compiled\_cond before each step, or use a context manager that the cross-attention queries.

Adding debug logs inside the UNet’s attention forward (or checking if ConditionCrossAttn.concat gets called) could help. Right now, the system completes without error, so it’s not crashing – it’s just not giving meaningful images. This strongly suggests a logical no-op on the conditioning rather than a hard failure. Once this injection is fixed, SDXL images should reflect the prompt.

## WAN 2.2 Pipeline Structure and Observations

The WAN v2.2 pipeline (for text-to-video and image-to-video) is handled somewhat differently, reflecting the complexity of video diffusion. Under apps/backend/engines/wan22/ we have separate engine classes for different model sizes (e.g. Wan225BEngine for 5B, Wan2214BEngine for 14B). The design pattern here tries to accommodate two execution modes:

* **Diffusers Pipeline Mode:** If a HuggingFace Diffusers model repository is available for WAN2.2, the engine will load a pre-built pipeline. In the code, during load(), it uses WanPipeline.from\_pretrained(...) along with a custom AutoencoderKLWan to initialize the entire pipeline[[14]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/wan22/wan22_5b.py#L116-L125)[[15]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/wan22/wan22_5b.py#L126-L134). If successful, self.\_comp.pipeline is set to this diffusers pipeline object. This pipeline internally includes its text encoder (a T5 or U-Net based text model for WAN), the temporal UNet, VAE, etc., all wired together.
* **Custom GGUF Execution Mode:** If no diffusers pipeline is found (e.g., the model is provided in a low-level .gguf format with separate components), the engine falls back to a manual implementation. The code checks if comp.pipeline is not None to decide the path[[16]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/wan22/wan22_5b.py#L149-L158). In pipeline mode, it calls \_run\_t2v / \_run\_i2v; in GGUF mode, it uses a runtime.nn.wan22 module. The GGUF path (apps/backend/runtime/wan22/wan22.py) is essentially a self-contained engine: it loads the **two-stage transformer models** (high-stage and low-stage) and runs the diffusion loop itself in Python. We see the engine assembling a RunConfig with all needed parameters (dimensions, fps, prompts, seed, etc.)[[17]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/wan22/wan22_5b.py#L169-L178)[[18]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/wan22/wan22_5b.py#L192-L201) and then calling gguf.run\_txt2vid(cfg) streaming events back to the UI[[19]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/wan22/wan22_5b.py#L208-L218). The runtime/wan22/wan22.py module is quite large, but in essence it manually performs:
* Text encoding via a T5-based model (the code ensures a text encoder path or HF tokenizer is available in resolve\_user\_supplied\_assets and config),
* High-stage diffusion (producing a coarse video latent),
* Low-stage diffusion (refining the latent sequence),
* Decoding to frames with the VAE.

**Prompt Injection in WAN 2.2:** In the diffusers pipeline case, the prompt is passed directly to the WanPipeline call, so we rely on that implementation to handle text encoding internally. For example, in run\_txt2vid, they simply do pipe(prompt=request.prompt, negative\_prompt=..., ...) and get output frames[[20]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/use_cases/txt2vid.py#L17-L26). We can assume the diffusers library’s WanPipeline correctly applies the text encoder, since it’s an official or reference implementation. Indeed, the code doesn’t manually manipulate conditioning here – it’s handled by the pipeline (which likely calls the WAN text encoder under the hood).

For the GGUF/manual mode, our code explicitly provides the prompt and negative prompt to the RunConfig[[21]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/wan22/wan22_5b.py#L174-L182). Inside wan22.py, those will be fed to the loaded tokenizer/text-encoder (either the one we loaded from HF assets or from the GGUF weights) to produce conditioning. The code is quite intricate, but given the effort put into this module, it likely does inject the text encoder output correctly (the WanDiTGGUF classes handle cross-attention with text, etc.). If there were an injection issue here, we would expect completely garbled video output – but since the user’s concern was specifically about SDXL, it sounds like WAN might be functioning closer to expected (perhaps due to relying on the tested diffusers pipeline or a more self-contained reference approach).

**Design Pattern Reflections:** The approach taken – having a common interface but multiple code paths (one for diffusers, one for GGUF) – has made the WAN 2.2 integration rather complex. However, unlike SDXL, in the diffusers mode the heavy lifting is done by an external, presumably correct pipeline. This may be why we don’t hear about prompt conditioning issues for WAN; the diffusers implementation would handle it. In the custom mode, the developers actually inlined a lot of logic (likely ported from the official WAN release) to ensure it works. This part of the code is advanced and less A1111-like, but it seems to adhere to the model’s needs as per the architecture “bible” (two stages, etc.).

One thing to note is that the **design pattern** overall (for both SDXL and WAN) is quite ambitious: it attempts to abstract engines and pipelines in a unified way (the Codex engines, use\_case runners, etc.). In practice, this led to a lot of moving parts and possibly confusion: - Some logic is split between engine classes (which prepare models, do minimal pre-processing) and the pipeline runner (which is more generic and tries to cover txt2img for any model). While modularity is good, it can make the flow of data harder to follow. For instance, the SDXL engine computes cond and uncond and then passes them down to the runner via global state (processing.sd\_model and function arguments). Debugging where exactly the prompt embedding goes into the UNet required tracing through multiple layers. - The **legacy vs new code** coexistence: there are .legacy references and this .sangoi folder with design notes. This indicates the design was evolving, potentially causing parts of the system to become out-of-sync. The SDXL pipeline might not have been fully validated after a refactor (for example, a recent refactor around 2025-10-31 mentions engine runtime contracts for SDXL).

**Suggestions to Improve/Resolve:**

1. **Verify and Fix Conditioning Injection in SDXL:** Focus on the StableDiffusionXL.get\_learned\_conditioning output usage. A straightforward way is to test a prompt and inspect the cond dict contents and ensure the UNet sees them. You might add logging inside ConditionCrossAttn or inside the UNet forward to confirm it's receiving a 2048-dim context. If it’s not, you may have to explicitly pass the conditioning to the UNet. For example, if using diffusers’ UNet2DConditionModel, one would call it with encoder\_hidden\_states=cond['crossattn'] and perhaps an extra text\_embeds=cond['vector'] for SDXL. Our code doesn’t do that explicitly, relying on patched mechanisms – which might need a tweak. Ensuring that runtime.unet.model (a KModel) is properly patched to use the y vector is crucial. If not, we could manually add a small hook: e.g., set unet.extra\_concat\_condition = cond['vector'] if the UnetPatcher supports an extra\_concat\_condition (there is an extra\_concat\_condition attribute in UnetPatcher[[22]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/patchers/unet.py#L62-L70)[[23]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/patchers/unet.py#L72-L75) that might be intended for exactly this purpose – to hold extra conditioning like ControlNet or global vectors).
2. **Simplify or Document the Pipeline Flow:** Given the complexity of the design pattern, consider documenting the end-to-end flow clearly (from request -> engine -> runner -> sampler -> model) in one of the .md files for future contributors (and your future self). This will make it easier to spot where data might be dropping out. Indeed, many .sangoi docs exist; updating them with the final design would help. If the design pattern feels overly convoluted (“didn’t work out”), you might even refactor to consolidate steps – for example, have the engine’s txt2img directly perform the steps instead of going through an indirection, at least for debugging purposes. This could localize the problem.
3. **Leverage Reference Implementations:** For SDXL, you have the option of using the HuggingFace diffusers pipeline as a reference to see how they handle dual encoders. You don’t necessarily want to use it in production (since you’ve rolled your own), but you can mimic their approach: they create prompt\_embeds and pooled\_prompt\_embeds and pass both to the UNet. If needed, you could call the diffusers StableDiffusionXLPipeline in a test to verify it produces correct images, then compare what conditioning it passes into its UNet vs what our Codex pipeline does. This comparison could highlight what’s missing (for example, maybe we aren’t normalizing or doing CLIP skip correctly – SDXL uses clip\_skip=2 by default as set in the spec[[24]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/sd/spec.py#L314-L322)[[25]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/sd/spec.py#L324-L331), which our code does handle by default to 2).
4. **Testing and Iteration:** Once changes are made (e.g., ensuring the y vector is injected), test the SDXL pipeline with a known prompt to see if the output changes from pure noise to a coherent image. SDXL outputs should be significantly better than noise even at 30 steps if conditioning works. Use a strong prompt and verify. You can also test extreme cases (like a blank prompt vs a detailed prompt) to ensure negative vs positive conditioning is doing something (blank prompt should yield only abstract noise, a detailed prompt should yield a discernible scene).
5. **WAN 2.2 Confidence:** The WAN pipeline appears to be in better shape likely because it either uses diffusers (which would have been tested externally) or a closely adapted version of the model’s official code. Focus on SDXL first. After fixing SDXL, run a quick check on WAN 2.2 as well (especially the GGUF mode) to ensure prompts influence the video (e.g., does a text prompt change the motion/scene as expected?). The code for WAN looks thorough, but double-check if the text encoder path is correctly resolved (the WanComponents includes text\_encoder and tokenizer directories, and RunConfig carries them[[26]](https://github.com/sangoi-exe/stable-diffusion-webui-codex/blob/5b5e993f4eba43c867f2a58fea7c985e5ff992c2/apps/backend/engines/wan22/wan22_5b.py#L176-L184)).

In conclusion, the **design pattern** here is powerful but intricate. The SDXL issue likely boils down to a small oversight in connecting the conditioned outputs to the UNet. By addressing that (ensuring cond actually conditions the model), you should see SDXL produce meaningful results. As a final note, consider gradually testing each pipeline component in isolation – for instance, directly call get\_learned\_conditioning and feed it into the diffusers UNet to see if images improve, etc. This stepwise testing can isolate the exact point of failure. Once fixed, your custom WebUI will benefit from the combined flexibility of A1111’s UI and the robust, properly-conditioned pipelines of SDXL and WAN 2.2.
